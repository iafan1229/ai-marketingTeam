- 관련 문서: [Codex Multi-Agent Working Plan](./codex-multi-agent-workflow.md)

## 1. 배경

헬스로그는 운동 기록 경험을 더 단순하고 빠르게 만들기 위한 앱이다. 현재 가장 큰 제약은 앱 자체의 품질만이 아니라, 사용자에게 지속적으로 앱의 존재와 가치를 알리는 마케팅 실행의 반복 비용이다.

혼자서 마케팅을 수행할 경우 다음 문제가 반복된다.

- 오늘 어떤 주제로 올릴지 매번 새로 고민해야 한다.
- 게시글 작성과 수정에 시간이 많이 든다.
- 어떤 글이 반응이 좋았는지 누적 학습이 어렵다.
- 사용자 피드백이 콘텐츠와 제품 개선으로 연결되지 않는다.

이 프로젝트의 목표는 단순한 "AI 글쓰기 기능"이 아니라, 헬스로그를 위한 작은 마케팅 팀을 소프트웨어로 만드는 것이다.

## 2. 문제 정의

현재 마케팅은 아래와 같은 비효율을 가진다.

- 전략 부재: 주제 선정이 즉흥적이다.
- 생산성 저하: 게시글 초안을 매번 처음부터 작성한다.
- 학습 부재: 성과 데이터가 다음 콘텐츠에 체계적으로 반영되지 않는다.
- 전환 단절: 공감, 댓글, 관심, 설치로 이어지는 흐름이 설계되어 있지 않다.

## 3. 비전

HealthLog AI Marketing Team은 다음 루프를 자동화하거나 반자동화한다.

`콘텐츠 기획 -> 콘텐츠 생성 -> 승인/발행 -> 반응 수집 -> 패턴 분석 -> 다음 콘텐츠 개선`

핵심 비전은 "노력형 마케팅"을 "시스템형 마케팅"으로 바꾸는 것이다.

## 4. 목표

### 제품 목표

- 헬스로그에 맞는 SNS 콘텐츠 주제를 빠르게 제안한다.
- 헬스로그 톤에 맞는 게시글 초안을 자동으로 생성한다.
- 게시 성과와 사용자 반응을 기록하고 다음 콘텐츠에 반영한다.
- 창업자 1인이 마케팅 팀처럼 운영할 수 있게 한다.

### MVP 목표

- 하루 1~3개의 게시 후보를 5분 안에 만들 수 있어야 한다.
- 사용자는 주제 추천, 글 생성, 결과 저장을 한 화면 또는 단순한 흐름에서 수행할 수 있어야 한다.
- 잘된 후킹 문구와 CTA 패턴을 최소 수준으로 누적할 수 있어야 한다.

### 비즈니스 목표

- 게시 빈도 증가
- 댓글/답글 기반 피드백 증가
- 앱 관심도 증가
- 자연 유입 다운로드 증가

## 5. 비목표

초기 버전에서 아래 항목은 제외한다.

- 완전 자동 무인 발행
- 복잡한 권한 체계
- 멀티 브랜드 지원
- 고도화된 광고 집행 자동화
- 다채널 대규모 운영
- 정교한 attribution 분석

## 6. 핵심 사용자

### 1차 사용자

- 헬스로그를 직접 만들고 운영하는 창업자 또는 1인 팀

### 2차 사용자

- 향후 다른 개인 앱/사이드 프로젝트 운영자

### 사용자 특성

- 제품 이해도는 높지만 마케팅 반복 작업은 피로함
- 빠르게 초안을 받고 직접 수정/승인하는 방식을 선호함
- 완전 자동화보다 "품질 좋은 반자동화"를 원함

## 7. Jobs To Be Done

- 오늘 올릴 만한 주제를 빨리 정하고 싶다.
- 헬스로그다운 말투로 글 초안을 받고 싶다.
- 댓글 유도형, 공감형, 업데이트형 콘텐츠를 상황에 맞게 만들고 싶다.
- 어떤 포맷이 먹히는지 쌓아두고 싶다.
- 제품 피드백과 마케팅을 하나의 루프로 묶고 싶다.

## 8. 제품 원칙

- 사람 승인 우선: 초기에는 사용자가 최종 승인한다.
- 짧고 자주: 장문 1개보다 짧은 실전 게시물을 우선한다.
- 제품 중심: 콘텐츠는 헬스로그 사용 경험과 개선 흐름을 반영한다.
- 학습 가능한 구조: 성과 데이터는 다음 생성에 반영할 수 있어야 한다.
- 단순한 운영: 1인 팀 기준으로 즉시 사용할 수 있어야 한다.

## 9. MVP 범위

MVP는 아래 3개 에이전트와 1개 사용자 역할로 구성한다.

### Planner Agent

역할:

- 오늘 올릴 콘텐츠 주제 제안
- 콘텐츠 목표 정의
- 추천 CTA 정의

입력:

- 최근 앱 업데이트
- 최근 사용자 피드백
- 최근 성과 요약

출력:

- 주제 3개
- 각 주제의 목표
- 각 주제의 CTA

### Copywriter Agent

역할:

- Planner가 선택한 주제를 바탕으로 실제 게시글 초안 생성

입력:

- 주제
- 목표
- CTA
- 톤 가이드

출력:

- 게시글 초안 2~3개
- 후킹 문구 후보
- 댓글 유도 질문

### Analyst Agent

역할:

- 저장된 게시 성과를 읽고 다음 전략에 반영할 인사이트 도출

입력:

- 게시글 텍스트
- 게시 후 성과 데이터

출력:

- 잘된 후킹 패턴
- 반응이 약한 표현
- 다음에 강화할 전략

### Manager (사용자)

역할:

- 주제 선택
- 생성 결과 검토
- 수정/승인
- 게시 후 성과 기록

## 10. 대표 사용자 흐름

### Flow A. 오늘의 게시글 만들기

1. 사용자가 앱 업데이트나 메모를 입력한다.
2. Planner Agent가 주제 3개를 제안한다.
3. 사용자가 주제 1개를 선택한다.
4. Copywriter Agent가 게시글 초안 2~3개를 만든다.
5. 사용자가 하나를 수정 후 저장한다.
6. 사용자가 수동으로 게시한다.

### Flow B. 어떤 글이 잘됐는지 확인하기

1. 사용자가 게시 후 성과 수치를 입력한다.
2. Analyst Agent가 최근 게시 성과를 요약한다.
3. 시스템이 다음 콘텐츠 생성 시 참고할 메모리를 갱신한다.

## 11. 기능 요구사항

### 콘텐츠 기획

- 사용자는 앱 업데이트, 피드백, 아이디어 메모를 입력할 수 있어야 한다.
- 시스템은 입력을 기반으로 주제 3개 이상을 제안해야 한다.
- 각 주제에는 목적과 CTA가 포함되어야 한다.

### 콘텐츠 생성

- 사용자는 선택한 주제로 글 초안 생성을 요청할 수 있어야 한다.
- 시스템은 짧은 SNS 게시물 초안 2~3개를 반환해야 한다.
- 결과에는 후킹 문구와 질문형 CTA가 포함되어야 한다.

### 성과 저장

- 사용자는 게시글별 성과를 기록할 수 있어야 한다.
- 최소 저장 항목은 플랫폼, 게시일, 텍스트, 좋아요, 댓글, 저장, 메모다.

### 분석

- 시스템은 최근 성과를 읽고 반복 가능한 인사이트를 생성해야 한다.
- 인사이트는 다음 콘텐츠 생성 시 참고 가능한 형태로 저장되어야 한다.

### 기록/메모리

- 시스템은 "잘된 후킹", "피해야 할 패턴", "추천 CTA"를 누적 저장해야 한다.

## 12. 비기능 요구사항

- 응답 속도: 단일 생성 요청은 체감상 10초 내외를 목표로 한다.
- 비용 효율성: 짧은 게시물 중심으로 운영하여 월 수천 원~수만 원 수준을 목표로 한다.
- 신뢰성: 생성 실패 시 재시도하거나 사용자에게 에러를 명확히 보여줘야 한다.
- 보안: API 키는 서버에서만 사용하며 클라이언트에 노출되지 않아야 한다.
- 확장성: 향후 발행 큐, 자동 예약, 채널 확장을 추가할 수 있어야 한다.

## 13. 제안 기술 구조

초기 기술 구조는 최대한 단순하게 유지한다.

- 프론트엔드: Next.js 관리자 UI
- 서버: Next.js Route Handler
- AI 호출: OpenAI Responses API
- 저장소: SQLite 또는 Postgres
- 에이전트 구조: 함수 단위 agent 모듈

예상 초기 엔드포인트:

- `POST /api/plan`
- `POST /api/generate`
- `POST /api/analyze`

예상 모듈 구조:

- `lib/agents/planner.ts`
- `lib/agents/copywriter.ts`
- `lib/agents/analyst.ts`

## 15. 성공 지표

### MVP 운영 지표

- 주간 생성 게시 후보 수
- 실제 게시 전환율
- 게시 준비 시간 단축
- 주간 피드백 수집량

### 콘텐츠 성과 지표

- 게시당 댓글 수
- 저장 수
- 프로필 방문 또는 앱 관심 표현 수

### 제품 지표

- 헬스로그 관련 콘텐츠 발행 빈도
- 콘텐츠 기반 기능 아이디어 반영 수
- 유입 설치 또는 관심 DM 증가

## 16. 출시 단계

### Phase 1. Manual Loop MVP

- 주제 추천
- 초안 생성
- 성과 수동 입력
- 분석 요약

### Phase 2. Operator Console

- 간단 관리자 화면
- 저장된 초안/성과 조회
- 메모리 재사용

### Phase 3. Semi-Automation

- 발행 큐
- 예약 발행
- 채널별 포맷 분기

## 17. 리스크

- 생성된 카피가 광고 문구처럼 느껴질 수 있다.
- 데이터가 적으면 Analyst 인사이트 품질이 낮을 수 있다.
- 지나친 자동화는 오히려 톤 일관성을 해칠 수 있다.
- 초기에 플랫폼별 실제 성과 데이터 연동이 제한될 수 있다.

## 18. 오픈 질문

- MVP에서 지원할 첫 플랫폼은 Threads만 할지, Instagram/ X까지 포함할지?
- 성과 입력은 수동으로 시작할지, 일부 API 연동까지 바로 할지?
- 헬스로그 브랜드 톤을 어떤 문장 예시로 고정할지?
- 게시 자동 발행은 언제부터 허용할지?

## 19. 현재 결론

이 프로젝트의 첫 버전은 "완전 자동 마케팅 봇"이 아니다.

첫 버전은 아래 성격을 가진다.

- 창업자 보조형
- 반자동 운영형
- 콘텐츠 중심
- 학습 가능한 마케팅 시스템

즉, `HealthLog AI Marketing Team`의 MVP는 "기획자 + 작가 + 분석가 + 사용자 승인" 구조의 경량 운영 시스템으로 정의한다.
