# Codex Multi-Agent Working Plan

## 문서 정보

- 문서명: Codex Multi-Agent Working Plan
- 상태: Draft v1
- 작성일: 2026-04-17
- 대상 프로젝트: HealthLog AI Marketing Team
- 문서 목적: Codex를 사용할 때 메인 에이전트와 서브 에이전트로 개발 작업을 어떻게 분리할지 정의

## 1. 문서 범위

이 문서는 `제품 내부 에이전트`를 설명하는 문서가 아니다.

여기서 말하는 에이전트는 아래 의미다.

- 메인 에이전트: 전체 개발 작업을 조율하는 Codex 주체
- 서브 에이전트: 메인 에이전트가 병렬로 위임하는 Codex 작업 단위

즉, 이 문서는 `개발 프로세스용 에이전트 분담 문서`다.

## 2. 제품 에이전트와의 구분

헷갈리지 않도록 용어를 분리한다.

### 제품 내부 에이전트

- `Planner Agent`
- `Copywriter Agent`
- `Analyst Agent`

이들은 나중에 HealthLog AI Marketing Team 제품 안에서 실제 기능으로 동작하는 런타임 구성요소다.

### Codex 작업 에이전트

- `Main Agent`
- `Sub Agent`

이들은 제품을 만들기 위해 Codex가 개발 작업을 나눠 수행할 때의 협업 구조다.

## 3. 목표

Codex 멀티 에이전트 작업 방식의 목표는 다음과 같다.

- 큰 작업을 병렬화해서 개발 속도를 높인다.
- 역할별 책임을 분명히 하여 충돌을 줄인다.
- 메인 에이전트가 통합 품질을 책임지게 한다.
- 서브 에이전트가 좁은 범위에 집중하도록 만든다.

## 4. 기본 원칙

- 메인 에이전트는 항상 전체 요구사항과 최종 결과 품질을 책임진다.
- 서브 에이전트는 명확한 입력, 출력, 파일 소유 범위를 가져야 한다.
- 같은 파일을 여러 서브 에이전트가 동시에 수정하지 않는다.
- 메인 에이전트는 병렬화 가능한 작업만 위임한다.
- 통합, 충돌 해결, 최종 검증은 메인 에이전트가 수행한다.

## 5. 역할 정의

## 5.1 Main Agent

메인 에이전트의 책임은 다음과 같다.

- PRD 해석
- 범위 확정
- 작업 분해
- 서브 에이전트별 소유 범위 지정
- 진행 상황 수집
- 결과 통합
- 테스트 및 최종 검토
- 사용자와의 커뮤니케이션

메인 에이전트는 직접 수행해도 되는 작업이 있더라도, 병렬화 가치가 큰 경우 서브 에이전트로 나눈다.

## 5.2 Sub Agent

서브 에이전트의 책임은 다음과 같다.

- 자신에게 할당된 한정 범위 작업 수행
- 지정된 파일/모듈 중심으로 수정
- 변경 내용과 영향 범위를 명확히 보고
- 통합이 필요한 지점과 리스크를 메인 에이전트에게 전달

서브 에이전트는 전체 아키텍처 재설계를 임의로 결정하지 않는다.

## 6. 권장 작업 분해 방식

HealthLog AI Marketing Team MVP에서는 아래 분해를 기본값으로 사용한다.

### Main Agent

담당:

- 전체 MVP 범위 조율
- API/DB/UI 연결 구조 결정
- 최종 통합
- 검증

### Sub Agent 1. API / Agent Logic

담당:

- `planner`
- `copywriter`
- `analyst`
- API route 구현

예상 소유 파일:

- `lib/agents/*`
- `app/api/*`

### Sub Agent 2. Data / Persistence

담당:

- DB 스키마
- 저장 로직
- 메모리 인사이트 저장 구조

예상 소유 파일:

- `lib/db.ts`
- `db/schema.*`
- `lib/repositories/*`

### Sub Agent 3. Admin UI

담당:

- 관리자 화면
- 주제 추천 요청 UI
- 초안 생성 UI
- 성과 입력 UI

예상 소유 파일:

- `app/page.tsx`
- `components/*`
- `app/(dashboard)/*`

### Sub Agent 4. QA / Docs

담당:

- 테스트 시나리오
- 수동 검증 체크리스트
- 개발 문서 업데이트

예상 소유 파일:

- `docs/*`
- `tests/*`

## 7. 단계별 운영 방식

### Phase 1. 설계

메인 에이전트가 수행:

- PRD 확인
- MVP 범위 확정
- 작업 분할
- 각 서브 에이전트 입력/출력 정의

이 단계에서는 메인 에이전트가 구조를 먼저 잡고, 애매한 범위를 줄인 뒤 위임한다.

### Phase 2. 병렬 구현

서브 에이전트가 수행:

- API 구현
- DB 설계
- UI 구현
- 테스트/문서 초안 작성

메인 에이전트가 수행:

- 중간 충돌 확인
- 인터페이스 정합성 확인
- 필요한 경우 작업 재분배

### Phase 3. 통합

메인 에이전트가 수행:

- 서브 결과 통합
- 타입/인터페이스 맞춤
- 상태 흐름 정리
- 리그레션 확인

### Phase 4. 검증

메인 에이전트 주도, QA 서브 에이전트 보조:

- 수동 테스트
- 에러 케이스 점검
- 문서 보완

## 8. 위임 규칙

메인 에이전트는 위임 시 아래 항목을 반드시 포함한다.

- 작업 목적
- 담당 범위
- 수정 가능한 파일 경로
- 수정하면 안 되는 영역
- 기대 출력물
- 보고 형식

좋은 위임 예시는 다음과 같다.

- "Sub Agent 2는 DB 스키마와 저장 로직만 담당한다."
- "UI 파일은 건드리지 않는다."
- "완료 시 수정 파일 목록과 통합 시 주의점을 함께 보고한다."

나쁜 위임 예시는 다음과 같다.

- "전체 프로젝트를 보고 적당히 구현해라."
- "필요한 건 다 바꿔라."

## 9. 파일 소유 원칙

파일 충돌을 줄이기 위해 초기에는 소유권을 강하게 나눈다.

예시:

- API 계층은 `Sub Agent 1`
- 데이터 계층은 `Sub Agent 2`
- UI 계층은 `Sub Agent 3`
- 문서/테스트는 `Sub Agent 4`

메인 에이전트만 여러 계층을 건드려 통합할 수 있다.

## 10. 커뮤니케이션 규칙

- 메인 에이전트는 사용자와 직접 소통한다.
- 서브 에이전트 결과는 메인 에이전트가 검토 후 반영한다.
- 사용자에게는 서브 에이전트 내부 세부 진행보다 통합 상태를 중심으로 공유한다.

## 11. 리스크

- 역할이 겹치면 같은 파일 충돌이 발생한다.
- 메인 에이전트가 범위를 좁혀주지 않으면 서브 에이전트 품질이 흔들린다.
- 너무 작은 작업까지 쪼개면 오히려 통합 비용이 커진다.
- 문서 없이 병렬화하면 아키텍처 일관성이 약해진다.

## 12. 현재 프로젝트에 대한 권장안

HealthLog AI Marketing Team의 첫 구현 단계에서는 아래 구성을 권장한다.

### 권장 구성 A. 1 Main + 3 Sub

- Main Agent: 전체 조율 및 통합
- Sub Agent 1: API / Agent Logic
- Sub Agent 2: Data / Persistence
- Sub Agent 3: Admin UI

이 구성이 가장 단순하고 현실적이다.

### 권장 구성 B. 1 Main + 4 Sub

- Main Agent: 전체 조율 및 통합
- Sub Agent 1: API / Agent Logic
- Sub Agent 2: Data / Persistence
- Sub Agent 3: Admin UI
- Sub Agent 4: QA / Docs

테스트와 문서를 병렬화해야 할 때 이 구성을 사용한다.

## 13. 첫 스프린트 분담안

첫 스프린트에서는 아래 수준으로 시작한다.

### Main Agent

- 프로젝트 초기 구조 결정
- 디렉터리 생성
- 인터페이스 계약 정리
- 결과 통합

### Sub Agent 1

- `POST /api/plan`
- `POST /api/generate`
- `POST /api/analyze`
- `lib/agents/planner.ts`
- `lib/agents/copywriter.ts`
- `lib/agents/analyst.ts`

### Sub Agent 2

- DB 선택
- 스키마 정의
- draft/result/insight 저장 로직

### Sub Agent 3

- 입력 폼
- 주제 추천 결과 목록
- 초안 생성 결과 화면
- 성과 수동 입력 화면

## 14. 문서 결론

Codex 멀티 에이전트는 제품 기능을 나누는 문서가 아니라, `제품을 만드는 개발 작업`을 나누는 문서다.

이 프로젝트에서는 아래 두 층을 항상 분리해서 이해한다.

- 제품 층: `Planner / Copywriter / Analyst`
- 개발 층: `Main Agent / Sub Agent`

현재 기준 최적의 작업 구조는 `1 Main Agent + 3 Sub Agents`이며, 필요 시 QA/Docs용 서브 에이전트를 추가한다.
